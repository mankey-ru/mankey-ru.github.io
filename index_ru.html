Адаптер CSM API

Критически важный бэкенд-сервис Node.js для РЖД CSM (Центр содействия мобильности пассажиров). Узел обработки четырех различных типов заявок маломобильных пассажиров со сложными моделями данных; интегрирует следующие системы:

  • Сайт РЖД https://www.rzd.ru/ru/9290 — Kafka
  • Административные интерфейсы РЖД — Kafka
  • АСУ ЕКМП (Единое клиентское мобильное приложение) — JSON REST API
  • АСУ ТТК (ТрансТелеКом) — JSON REST API
  • АСУ СИОП ФПК (Система идентификации и обслуживания пассажиров, Федеральная пассажирская компания) — JSON REST API

Двунаправленная синхронизация на всем жизненном цикле заявки, уведомления пользователям и администраторам и многое другое.

Информаторы версий (IVI + RVI)

Два бэкенд-сервиса: IVI (сторона компании) и RVI (развернут на сервере заказчика — ОАО «РЖД»). Сервисы полностью автоматизируют жизненный цикл выпуска ПО — от приема запроса на слияние (merge request) до публикации релиза и установки в цепочке серверных окружений (от тестовых до промышленного).

Ключевые возможности и обязанности:

  • Разработал и сопровождал интеграции с: Wflow (трекер задач), MCC (менеджер установки), RVI, PostgreSQL, FTP, SVN, GitLab, Mattermost и Kibana.
  • Реализовал структурированное логирование в Elasticsearch с мониторингом в реальном времени.
  • Настроил интеллектуальные уведомления в Mattermost по нескольким каналам, включая ссылки на формы повторного запуска, логи Kibana и сведения об ошибках.
  • Достиг высокого покрытия тестами с помощью комплексных end-to-end и модульных тестов (Jest).
  • Обработчик приема MR: Проверяет поля задачи в трекере, проверяет на дубликаты, собирает данные контракта/эпика, определяет измененные репозитории, записывает данные в базу IVI и передает их в RVI через REST API.
  • Обработчик публикации релиза: Обнаруживает «потерянные» задачи, скачивает и распаковывает дистрибутив, генерирует артефакты для заказчика (6 типов документов) и JSON состава релиза для надежной передачи данных в RVI (резервный вариант при недоступности REST API).
  • Обработчик событий установки: Фиксирует успешные установки/откаты на сервере, выполняет массовое обновление статусов релизов и связанных задач через Bulker API, добавляет комментарии об откате.

Пояснение форматов метаданных
Что такое метаданные для административных форм

Эти файлы метаданных представляют собой декларативные JSON-дескрипторы, определяющие полный жизненный цикл административной формы - от базовой таблицы (таблиц) базы данных до поведения UI в браузере.

Это устраняет повторяющийся CRUD-шаблонный код, обеспечивая согласованность для более чем 1000 форм.

Каждый дескриптор сопоставляет столбцы базы данных (или межтабличные связи) с определениями полей, задающими поведение в трех контекстах:

  • Представление списка
    • Фильтрация
    • Сам список (отображение, сортировка, редактирование записей на лету)
  • Представление формы (создание, редактирование одной записи)

Каждое поле формы может иметь разные типы элементов управления во всех трех контекстах, каждый со своими параметрами, такими как валидация, значения по умолчанию или пользовательский шаблон. Примеры типов элементов управления:

  • Поле ввода (текст, целое, с плавающей точкой, email и т.д.) или текстовая область
  • Выпадающий список (вкл. асинхронный поиск), всплывающий выбор, оба могут использоваться для множественного выбора записей
  • Дата с временем или без, в контексте фильтра может использоваться как диапазон
  • Выбор на карте или выбор адреса (геокодер)
  • Редактор графиков
  • ...и некоторые другие

Что такое метаданные для пользовательских страниц

Эти файлы метаданных страниц определяют полную HTML-страницу или слой получения данных (на базе JSON), устраняя необходимость в пользовательском бэкенд-коде для каждой страницы (всего: 500+ страниц).

Компоненты — это гранулярные строительные блоки, которые:
  • Перечисляют точные поля для выборки
  • Назначают первичные ключи
  • Определяют связи (один-к-одному, один-ко-многим)
  • Обрабатывают пагинацию и фильтрацию
