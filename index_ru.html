Адаптер CSM API

Критически важный бэкенд-сервис на Node.js для CSM РЖД (Центр содействия мобильности). Хаб для обработки четырёх типов заявок пассажиров с ограниченными возможностями со сложными моделями данных; интегрирует следующие системы:

  • Сайт РЖД https://www.rzd.ru/ru/9290 — Kafka
  • Административные интерфейсы РЖД — Kafka
  • ЕКМП АСУ (Единое клиентское мобильное приложение) — JSON REST API
  • ТТК АСУ (ТрансТелеКом) — JSON REST API
  • СИОП ФПК АСУ (Система идентификации и обслуживания пассажиров, Федеральная пассажирская компания) — JSON REST API

Двусторонняя синхронизация на всех этапах жизненного цикла заявки, уведомления пользователям и администраторам и многое другое.

Информеры версий (IVI + RVI)

Два бэкенд-сервиса: IVI (на стороне компании) и RVI (развёрнут на сервере заказчика — РЖД). Сервисы полностью автоматизируют жизненный цикл выпуска ПО — от принятия запроса на слияние до публикации релиза и установки по всей цепочке серверных окружений (от staging до production).

Ключевые возможности и обязанности:

  • Разработал и поддерживал интеграции с: Wflow (трекер задач), MCC (менеджер установок), RVI, PostgreSQL, FTP, SVN, GitLab, Mattermost и Kibana.
  • Реализовал структурированное логирование в Elasticsearch с мониторингом в реальном времени.
  • Настроил интеллектуальные уведомления в Mattermost по нескольким каналам, включая ссылки на формы повторного запуска, логи Kibana и детали ошибок.
  • Обеспечил высокое покрытие тестами с помощью комплексных end-to-end и модульных тестов (Jest).
  • Обработчик принятия MR: Проверяет поля задачи в трекере, ищет дубликаты, собирает данные по контрактам/эпикам, определяет изменённые репозитории, записывает данные в базу IVI и передаёт их в RVI через REST API.
  • Обработчик публикации релиза: Обнаруживает “потерянные” задачи, загружает и распаковывает дистрибутив, генерирует артефакты для заказчика (6 типов документов) и JSON состава релиза для надёжной передачи данных в RVI (запасной вариант при недоступности REST API).
  • Обработчик событий установки: Фиксирует успешные установки/откаты на сервере, выполняет массовое обновление статусов релизов и связанных задач через Bulker API, добавляет комментарии об откате.

Объяснение форматов метаданных
Что такое метаданные для админ-форм

Эти файлы метаданных представляют собой декларативные JSON-описания, которые определяют весь жизненный цикл админ-формы — от лежащих в основе таблиц базы данных до поведения UI в браузере.

Это устраняет повторяющийся CRUD-бойлерплейт, сохраняя согласованность для более чем 1000 форм.

Каждый дескриптор сопоставляет столбцы базы данных (или межтабличные связи) с определениями полей, задающими поведение в трёх контекстах:

  • Представление списка
    • Фильтрация
    • Сам список (отображение, сортировка, редактирование записей на лету)
  • Представление формы (создание, редактирование одной записи)

Каждое поле формы может иметь разные типы элементов управления во всех трёх контекстах, каждый со своими параметрами, такими как валидация, значения по умолчанию или пользовательский шаблон. Примеры типов элементов управления:

  • Input (текст, целое число, число с плавающей точкой, email и т.д.) или Textarea
  • Выпадающий список (включая асинхронный поиск), Popup Picker, оба могут использоваться для множественного выбора записей
  • Дата с временем или без, в контексте фильтра может использоваться как диапазон
  • Выбор карты или адреса (геокодер)
  • Редактор диаграмм
  • ...и некоторые другие

Что такое метаданные для пользовательских страниц

Эти файлы метаданных страниц определяют полную HTML-страницу или слой получения данных (управляемый JSON), устраняя необходимость в кастомном бэкенд-коде для каждой страницы (всего: более 500 страниц).

Компоненты — это гранулярные строительные блоки, которые:
  • Перечисляют конкретные поля для выборки
  • Указывают первичные ключи
  • Определяют связи (один-к-одному, один-ко-многим)
  • Обрабатывают пагинацию и фильтрацию
